1. Technical Field

This invention relates to machine-learning ranking systems and data-ingestion systems, and more specifically to adaptive score routing, segment-specialized model arbitration, explainable cohort-aware model selection, conservative relationship-aware dataset merge planning, and guarded upload-time numeric quantization in ranking workflows.

2. Technical Problem

Conventional lead scoring systems typically apply a single model to all incoming records, even when real-world outcomes differ materially across segments. Conventional data preparation pipelines also perform brittle joins or uncontrolled compression that can silently corrupt model inputs. Existing systems usually do not provide deterministic, inspectable routing policies, explainable arbitration ledgers, conservative join-shape enforcement, or distortion-guardrailed compression fallback modes. As a result, such systems are hard to trust, hard to govern, and weakly differentiated.

3. Invention Summary

Lucida introduces a routed adaptive ranking architecture in which a base ranking model may coexist with multiple segment-specialized models derived from real-world feedback. At score time, the system evaluates whether an incoming row matches one or more specialized cohorts, computes a deterministic routing priority, selects the winning model according to a defined arbitration policy, scores the row through the selected model, and emits a routing ledger describing policy, candidates considered, reason for selection, and selected model.

The invention further includes: (i) conservative relationship-aware merge planning that identifies candidate join keys and enforces join-shape safety, including refusal of many-to-many joins; (ii) upload-time numeric quantization with distortion guardrails, shadow mode, and safe-default execution mode; (iii) feedback matching through deterministic row signatures; and (iv) version-aware ranking comparison using prior artifact versions.

4. Core Inventive Concepts

  1. Adaptive generation of segment-specialized models from feedback events tied to prior scored rows.
  2. Score-time routing between a base model and one or more specialized models based on cohort matching.
  3. Deterministic arbitration among multiple matching specialized models using weighted operational signals including feedback volume and performance metrics.
  4. Emission of an inspectable routing ledger containing policy identifier, route reason, selected model, matched cohort, and candidate arbitration details.
  5. Deterministic feedback matching via row signatures linking score-time rows and later outcome uploads.
  6. Conservative relationship-aware merge planning and execution with join safety controls and one-to-many aggregation handling.
  7. Upload-time numeric quantization with measured distortion thresholds and execution-mode fallback behavior.

5. Independent Claim Drafts (Updated)

Claim Set A: Routing and Arbitration Method

1. A computer-implemented method for adaptive ranking of records, comprising: storing a base ranking model; storing a plurality of segment-specialized ranking models each associated with a respective segment definition; receiving an input record for scoring; determining whether the input record matches one or more of the segment definitions; when the input record matches multiple segment definitions, computing a priority value for each corresponding segment-specialized ranking model according to an arbitration policy; selecting, based on the priority values, one of the segment-specialized ranking models or the base ranking model; generating a ranking score for the input record using the selected model; and outputting, together with the ranking score, routing metadata identifying the selected model and a reason for its selection.

2. The method of claim 1, wherein the priority value is based on at least one of: feedback volume associated with the segment-specialized ranking model, an accuracy metric associated with the segment-specialized ranking model, and an area-under-curve metric associated with the segment-specialized ranking model.

3. The method of claim 1, wherein the routing metadata further comprises a policy identifier and a list of candidate segment-specialized ranking models considered during arbitration.

4. The method of claim 1, further comprising generating the segment-specialized ranking models by retraining on feedback events associated with previously scored records belonging to corresponding segments.

5. The method of claim 1, wherein the routing metadata is persisted with each scored record and includes candidate priority scores computed by a deterministic weighted function.

Claim Set B: Ranking System

6. A ranking system comprising: one or more processors; memory storing instructions; a feedback event store linking historical score outputs to subsequent outcomes; a model store containing a base model and one or more segment-specialized models; and a routing engine configured to match an incoming row against segment criteria, arbitrate between eligible models, select a winning model, produce a score, and produce a route explanation ledger.

7. The system of claim 6, wherein the route explanation ledger includes at least: a selected model identifier, a route type, a matched segment value, a policy identifier, a route reason, and one or more alternative candidate models not selected.

8. The system of claim 6, wherein the routing engine falls back to the base model when no specialized segment model matches the incoming row.

Claim Set C: Feedback-Driven Governance

9. A computer-implemented method comprising: storing scored row snapshots with deterministic row signatures; later receiving outcome data; matching the outcome data back to the scored row snapshots using the row signatures; calculating feedback performance metrics; identifying one or more drifting segments based on disparity between predicted score and actual outcome rate; and retraining a segment-specialized model using feedback rows belonging to a selected drifting segment.

10. The method of claim 9, further comprising automatically triggering retraining when a policy threshold is satisfied by at least feedback volume and one or more post-score performance indicators.

11. The method of claim 9, wherein the retrained segment-specialized model is subsequently used by the routing engine of claim 6 for future rows matching the selected drifting segment.

Claim Set D: Relationship-Aware Merge Planning

12. A computer-implemented method for preparing machine-learning input datasets, comprising: profiling columns across a plurality of uploaded datasets; generating pairwise candidate join keys based on at least name similarity, normalized name similarity, value overlap, normalized value overlap, statistical similarity, and coverage; assigning confidence scores to candidate join keys; selecting a join plan according to confidence and coverage thresholds; determining join-shape cardinality for selected joins; and executing only join operations that satisfy predefined safety constraints.

13. The method of claim 12, further comprising refusing a proposed merge when the join-shape cardinality is many-to-many or when no candidate join satisfies minimum confidence and minimum coverage thresholds.

14. The method of claim 12, wherein one-to-many joins are transformed by aggregation before merge execution to preserve base-row semantics for scoring.

Claim Set E: Guardrailed Upload Quantization

15. A computer-implemented method for preparing uploaded tabular data for ranking, comprising: identifying protected columns to remain lossless; identifying eligible numeric columns for quantization; applying randomized sign and permutation transforms to eligible numeric columns; quantizing transformed values to fixed-bit representations; reconstructing dequantized values; computing distortion metrics including reconstruction mean-squared error and inner-product error; and selecting an execution mode according to distortion thresholds.

16. The method of claim 15, wherein the execution mode is one of: full-precision bypass mode, shadow mode with quantization diagnostics while preserving full-precision scoring, and compressed execution mode when distortion metrics satisfy policy thresholds.

17. The method of claim 15, wherein the protected columns include at least one of: identifier-like columns, datetime-like columns, binary columns, target columns, and non-numeric columns.

Claim Set F: Version-Aware Ranking Comparison

18. A computer-implemented method comprising: storing versioned model artifacts for a model identifier; scoring a batch of rows with a current version; loading a prior artifact version; re-scoring the batch with the prior artifact version; and emitting per-row rank movement metadata including current rank, previous rank, rank delta, score delta, and comparison baseline timestamp.

19. The method of claim 18, wherein per-row alignment between current and prior scores is determined using deterministic row signatures derived from normalized row payloads.

6. Strong Dependent Claim Directions

  • Priority score weighted as a combination of recent feedback volume, ROC AUC, and accuracy.
  • Segment definitions derived from low-cardinality categorical fields identified from feedback event payloads.
  • Routing metadata displayed in a user interface as part of a scored-record explanation panel.
  • Arbitration details limited to top candidate models ranked by deterministic priority.
  • Automatic creation of a segment model name derived from a base model name and a segment key/value pair.
  • Use of versioned artifacts so routed decisions can later be compared against prior model generations.
  • Join confidence thresholds implemented as configurable policy controls for safe auto-merge.
  • Many-to-many merge refusal with explicit warning telemetry in merge-plan output.
  • Quantization protection rules that preserve identifier, temporal, and target semantics.
  • Distortion-threshold guardrails with fallback to full-precision execution when safety criteria fail.

7. Novelty and Differentiation Positioning

The strongest patentable center here is not "using ML to score leads." That is generic. The stronger inventive center is:

  1. A closed-loop system that converts outcome feedback into segment-specialized model variants.
  2. A score-time arbitration engine that decides which variant should handle a row.
  3. An inspectable routing ledger that makes the arbitration itself part of the output.
  4. A conservative relationship-aware merge planner that controls join safety before model execution.
  5. A guardrailed quantization path that trades memory and latency against bounded distortion under explicit policy.

This is stronger because it ties adaptation, selection, explainability, data-shape governance, and compression safety into one governed technical mechanism. That is more concrete than generic AutoML or ensemble scoring claims.

8. Commercially Useful Claim Framing

For usability, users do not need to know which segment model exists in advance, which join key should be used, or whether compression is safe for a given upload. The system determines route, merge strategy, and compression execution mode while emitting inspectable diagnostics. For patent framing, this supports a user-facing improvement in trustworthy adaptive scoring and safe data preparation, not merely an internal analytics pipeline.
